# Eros Tower 1.2.4

## 배포 범위

- 일반 업데이트 채널: `ErosTower.v1.update.js` / `1.2.4`
- 로컬 설치 파일: `☸에로스 타워.js` / `1.2.4`
- 기존 4.x 업데이트 호환 채널: `ErosTower.update.js` / `4.0.32`
- 작업 범위는 에로스 타워 플러그인과 릴리스 파일뿐이며 PocketRisu는 변경하지 않았다.

## 확인한 근원

1. 이미지 계획이 여러 선택·증거·컴파일 단계를 거치며 최종 출력보다 로컬 추론이 앞설 수 있었다.
2. 앨범 바인딩 단계가 플래너가 작성한 제목과 글귀를 버리고 `장면 S#` 제목 및 선택된 번역 문단으로 강제 교체했다.
3. 같은 턴의 여러 삽화는 한 플래너 응답에서 계획되지만, 동일 인물의 안정 외형 태그를 각 장면에서 같은 순서로 반복하라는 계약이 부족했다. NAI 전송 자체는 이미지별 캐릭터 프롬프트를 분리할 수 있었으나 입력 태그가 흔들리면 그대로 다른 외형이 됐다.
4. 누락된 앨범 메타데이터를 `장면 N`으로 대신 만드는 일반 fallback도 남아 있었다.
5. 첫 메시지, 예시 대화, Risu 요청 메시지, 실제 저장 채팅의 권위가 섞여 첫 장면과 현재 대화가 분리될 수 있었다.
6. 로어/CBS 활성화와 현재 대화 주입이 호스트의 실제 변수·키·저장 대화 상태보다 로컬 수집 결과에 치우칠 수 있었다.
7. 응답 직후 아직 저장되지 않은 assistant 본문을 상태에 먼저 커밋하면 이어쓰기·재생성·교체·재시작 시 상태와 실제 채팅이 어긋날 수 있었다.
8. 장기기억 원문을 짧게 잘라 저장하고 성장 중인 마지막 chunk를 고정하면 오래된 대화의 세부와 새로 붙은 꼬리가 사라질 수 있었다.

## 변경

### 첫 메시지와 현재 대화 권위

- Risu 요청 메시지와 실제 저장 채팅을 역할·ID·내용 기준으로 합쳐 현재 사용자/assistant 대화를 권위 자료로 사용한다.
- 처리된 `NewChat`, `fmIndex`, 등록 첫 메시지의 우선순위를 복구해 시작 장면을 유지한다.
- 예시 대화는 말투·형식 참고 자료로만 포장하고 현재 사건이나 정사 기억으로 커밋하지 않는다.
- 메인 주입에는 현재 대화 블록을 별도 예산으로 보존하고, 여러 pre-agent 초안을 중복 주입하지 않고 성공한 Synthesis 결과만 전달한다.

### 로어/CBS와 canonical 관점

- Risu형 조건문·변수·directive·확률·selective key·재귀·sticky 상태를 평가하고 활성 로어만 예산 안에서 packing한다.
- 시스템에 이미 주입된 NPC 색인이나 로어 본문을 현재 대화의 인물 등장 증거로 다시 사용하지 않는다.
- canonical annotation의 `knowledgeBoundary`, `futureSource`, `knownBy`, `cannotKnow`, route, reveal 조건을 runtime unit에 다시 합성해 비공개·미래정보 관점을 유지한다.

### 장기기억과 저장 assistant 커밋

- 장기기억 chunk는 원문과 줄바꿈을 자르지 않고 보존하며, 시작 위치 기반 ID로 같은 구간을 안정적으로 갱신한다.
- 성장 중인 마지막 chunk가 바뀌면 이전 추출 상태를 폐기하고 새 hash로 재추출한다. 삭제·재색인 시 이전 원문은 격리 보관한다.
- 상태 커밋은 호스트에 실제 저장된 assistant를 확인한 뒤 실행한다. 이어쓰기 suffix, 재생성 교체, 재시작 미처리 응답을 구분한다.
- assistant 교체는 검증된 이전 prefix snapshot으로 되돌린 뒤 새 assistant만 재생한다. 호환 snapshot이 없으면 잘못된 상태를 진행시키지 않고 진단에 남긴다.

### 단일 단계 이미지 계획

- 현재 최종 출력만 `[S#]` 단위로 한 번 계획하고 그 결과를 바로 이미지 API로 전달한다.
- 선택·증거·인물 카탈로그·샷별 재컴파일 단계와 생성 이미지 이력 재주입을 사용하지 않는다.
- 선택된 한 세그먼트만 현재 인물, 행동, 복장, 부상, 장소, 소품, 음식, 날씨를 결정한다.
- 최대 삽화 수가 아니라 최소 삽화 수를 사용하며, 존재하는 서로 다른 세그먼트 범위 안에서 최소 수를 맞춘다.

### NovelAI 캐릭터 프롬프트

- NAI V4는 장면 공통 base caption과 인물별 `char_captions[]`를 분리한다.
- 캐릭터 순서와 negative의 `sourceIndex`를 맞추고, 빈 캐릭터 negative 슬롯도 보존한다.
- `use_coords=false`, `use_order=true`를 사용하며 로컬 위치 격자나 좌표 추론을 다시 넣지 않는다.
- 같은 턴에서 동일 인물은 이름 또는 역할 표식과 안정 외형 태그 블록을 모든 샷에 같은 순서로 반복한다. 장면에 따라 바뀌는 표정·시선·자세·행동만 다시 기술한다.

### 앨범 제목과 글귀

- 스토리 레이아웃 바인딩은 `segmentId`와 문단 위치만 연결하고 플래너의 `title`, `memoryLine`, `summary`를 보존한다.
- `장면 S#` 강제 제목과 선택 문단 통삽입 함수를 삭제했다.
- 내장 플래너의 제목과 글귀는 각각 필수이며 40자/160자 이내의 새 한국어 앨범 문구로 작성한다. 누락되거나 지나치게 길면 로컬에서 문단을 잘라 넣지 않고 플래너 재시도로 돌린다.
- 일반 `장면 N` fallback도 제거했다.
- 이미 저장된 기존 앨범 항목은 자동 변조하지 않으며 신규 생성분부터 적용한다.

### API 응답 호환

- OpenAI 공식 API와 GPT-5/o-series 호환 모델에는 `max_completion_tokens`만 보내고, 기존 호환 서버에는 `max_tokens`를 유지한다.
- 이미지 플래너의 텍스트 추출은 일반 응답 본문을 우선하며 이미지 에이전트에서만 `reasoning_content`/`reasoning` 호환 경로를 사용한다.
