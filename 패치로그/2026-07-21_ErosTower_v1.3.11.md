# Eros Tower 1.3.11 — 장기 세션 canonical blob 증식 차단

## 관측과 원인

- 1.3.9 적용 후 canonical blob 수가 한 진행에서 149→172(+23), 다음 진행에서 172→190(+18)으로 늘었다.
- 1.3.9의 내용 기반 청크는 전체 약 170개 재생성을 막았지만, 채팅에 영구 ID가 없을 때 사용하는 fallback 스코프가 `updatedAt`을 해시에 포함했다.
- 따라서 같은 채팅도 매 진행마다 새 스코프로 인식됐다. 변경되지 않은 청크는 공유됐지만, 실제로 바뀐 18~23개 청크의 이전 세대는 과거 스코프가 계속 참조해 GC 대상이 되지 않았다.
- 이 패턴은 1.3.8에서 거의 전체 청크 수만큼 증가하고, 1.3.9에서 실제 변경 청크 수만큼만 증가한 실측과 일치한다.

## 변경 사항

- fallback 채팅 ID에서 `date`·`updatedAt`·변경 가능한 제목을 제외하고, 채팅/캐릭터 인덱스와 생성 시각·도입부 지문으로 안정적인 ID를 만든다.
- RisuAI `setChatToIndex`를 이용해 `erosTowerChatId`를 채팅에 한 번 저장하고 이후 모든 진행에서 같은 스코프를 사용한다.
- 기존 장기 세션은 현재 채팅 이력과 prefix가 일치하는 스코프 가운데 가장 강하게 검증되고 가장 긴 최신 상태를 선택하므로 상태·기억·canonical cache가 초기화되지 않는다.
- 저장 API가 없거나 ID 저장 확인에 실패해도 같은 이력의 기존 스코프를 다시 선택하거나 안정 fallback을 사용해 턴별 증식이 재발하지 않는다.
- 메시지 ID까지 정확히 일치한 동일 채팅의 과거 임시 스코프만 삭제한다. 내용만 우연히 같은 약한 후보와 다른 채팅 스코프는 보존한다.
- 여러 과거 스코프의 키를 한 번의 storage inventory로 일괄 제거하고 canonical GC도 마지막에 한 번만 수행해 정리 과정의 메모리·I/O를 제한한다.
- canonical GC는 참조 cache 수, blob 수, 보존/삭제 수, skip 여부와 실패한 cache key를 진단 정보에 기록한다. 참조를 읽지 못한 경우에는 기존처럼 삭제를 중단한다.

## 구조 안전성

- 기존 스코프를 선택한 뒤 해당 상태를 먼저 읽고, 현재 스코프는 정리 대상에서 명시적으로 제외한다.
- 강한 메시지 ID 지문이 없는 후보는 자동 삭제하지 않는다.
- 다른 캐릭터·다른 대화 이력의 스코프는 정리하지 않는다.
- 채팅 저장은 기존 객체를 펼쳐 복제하고 `erosTowerChatId` 한 필드만 추가한다.
- canonical blob 삭제는 모든 남은 `canonical-cache:*` 참조를 읽은 경우에만 수행한다.

## 검증

- 200개 전체 테스트, boolean 1,678개, true 1,655, 기존 진단 false 23, 오류 0.
- 신규 검증 14개 조건 통과:
  - `updatedAt` 변경 시 구 fallback ID가 달라지는 신고 조건 재현
  - 새 fallback ID 안정성
  - 최신 호환 스코프 선택 및 상태 연속성
  - 채팅 ID 저장 및 다음 턴 재사용
  - 과거 동일 채팅 스코프만 삭제
  - 현재 스코프와 무관한 채팅 스코프 보존
  - 과거 canonical 세대 회수 및 현재 세대 보존
  - canonical cache 읽기 실패 시 GC 안전 중단·실패 key 진단·blob 보존
- `node --max-old-space-size=64 --check` 통과.
- ESLint 8.57.1 `no-redeclare`, `no-dupe-keys` 통과.
- `git diff --check` 통과.
