# Eros Tower 1.2.9

## 증상

- 1.2.8 신규 기본 OFF 상태에서 으아악씨를 ON으로 저장해도 채팅 입력창의 런처 버튼이 나타나지 않을 수 있었다.
- 1.2.8은 플러그인 활성 완료 뒤에도 시작용 전체 설정을 읽으면서 40개 레거시 플러그인 인자 브리지 호출을 실행했다.

## 원인

- 1.2.8은 두 입주민이 모두 OFF면 시작 런처 동기화를 반환하면서 Risu `getRootDocument()` 호출도 함께 생략했다.
- Risu API의 `getRootDocument()`는 단순 DOM 조회가 아니라 `mainDom` 권한 확인과 메인 문서 프록시 획득을 함께 수행하며, 권한이 없으면 `null`을 반환한다.
- 최초 메인 DOM 권한/루트 획득이 전체화면 플러그인 설정창 안에서 ON을 저장하는 시점으로 밀렸고, 실패 결과는 저장 성공 메시지 뒤에 숨었다.
- 시작 동기화는 버튼 상태에 필요한 저장 설정 외에 40개 레거시 인자까지 읽었다.

## 수정

- 플러그인 초기화 단계에서 Risu 메인 DOM 권한과 루트를 정확히 한 번 확보하고 두 인채팅 런처가 공유한다.
- 두 입주민이 OFF면 버튼, 이벤트 리스너, MutationObserver는 만들지 않는다.
- 이후 으아악씨 또는 모시모씨를 ON으로 저장하면 이미 확보된 루트에 즉시 한 번 장착한다. 재시도 타이머나 폴링은 추가하지 않았다.
- 시작 런처 설정은 저장된 설정만 읽고 레거시 플러그인 인자 오버라이드는 읽지 않는다. 일반 요청과 설정 화면의 기존 전체 설정 해석은 그대로 유지한다.
- 1.2.8의 `pipelineJson` 전체 에이전트 저장 핫픽스는 유지한다.

## 배포 버전

- 일반 업데이트 / 직접 설치: `1.2.9`
- 기존 4.x 호환 업데이트 채널: `4.0.37`

## 검증

- 세 배포 파일 `node --check`: 통과
- ESLint 8 (`no-undef`, `no-redeclare`, `no-dupe-keys`, `Buffer` 전역 허용): 오류 0
- 내장 회귀 테스트: 193개, 실행 오류 0개
- 단언 기준선: 1.2.8의 기존 거짓 23개 동일, 신규 계약 2개 통과, 신규 실패 0개
- OFF 시작 재현: `getRootDocument` 1회, 플러그인 인자 0회, 버튼/리스너/옵저버 0개
- 으아악씨 ON 저장 재현: 추가 `getRootDocument` 호출 0회, 버튼 1개 즉시 장착, 옵저버 1개와 입력 이벤트 리스너 4개를 ON 이후에만 생성
- 동일한 인공 호스트 지연 시험: 1.2.7 최종 테스트본 943.1ms에서 1.2.9 287.2ms로 감소 (`69.5%`). 절대 시간은 시험용이며 실제 환경에 따라 달라진다.
- v1과 직접 설치 파일 SHA-256 동일, 호환 채널은 상단 5줄을 제외한 본문 동일
