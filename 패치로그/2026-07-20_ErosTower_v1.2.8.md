# Eros Tower 1.2.8

## 증상

- `에이전트 설정 전체 저장`을 누르면 성공 메시지가 표시되지만 창을 다시 열거나 플러그인을 다시 시작했을 때 에이전트 ON/OFF, 모델, 토큰 설정이 기존 값으로 돌아갈 수 있었다.
- 1.2.7은 플러그인 활성화 단계가 1.2.6보다 체감상 느려질 수 있었다.

## 원인

- 전체 저장 경로는 UI에서 읽은 에이전트 배열을 런타임 `pipeline`에만 넣었다.
- 저장 직전 `configForStorage()`는 파생 런타임 필드인 `pipeline`을 제거하므로, 실제 저장 권위인 `pipelineJson`은 이전 값으로 남았다.
- 1.2.7의 UI 등록은 전체 `getConfig()`와 인채팅 런처 동기화를 기다렸다. 이 과정에서 40개의 플러그인 인자를 순차 조회했다.
- 배포 스크립트 크기도 1.2.6의 2,403,247바이트에서 1.2.7의 3,942,551바이트로 증가해 파싱 부담이 늘었다.

## 수정

- 전체 저장 시 모든 에이전트 설정을 `pipelineJson`으로 직렬화한 뒤 정규화된 런타임 `pipeline`을 다시 만든다.
- 저장용 설정에서는 런타임 `pipeline`만 제거하고 새 `pipelineJson`을 보존한다.
- 저장 후 다시 파싱했을 때 에로스·Psyche·입주민 에이전트의 ON/OFF, Provider, 모델, 추론 단계, 온도, 토큰, 컨텍스트, 타임아웃과 실행 옵션이 유지된다.
- UI 버튼 등록은 더 이상 전체 설정 조회나 런처 동기화를 기다리지 않는다.
- 40개 플러그인 인자는 `Promise.all`로 함께 읽고 저장 설정 읽기와도 병렬 처리한다.
- 선택형 인채팅 런처 동기화는 플러그인 등록 이후로 지연하며, 으아악씨와 모시모씨가 모두 OFF면 DOM 런처 초기화를 건너뛴다.
- unload 시 아직 시작하지 않은 지연 동기화를 취소한다.

## 기본 설정 정책

- 신규 설치는 관리형 프롬프트 캐싱, 으아악씨, 모시모씨가 기본 OFF다.
- 기존 사용자가 저장한 명시적 ON/OFF는 그대로 보존한다.
- 기본 OFF 자체는 시작 지연의 원인이 아니며, 두 입주민이 OFF면 오히려 런처 작업을 생략한다.
- 저장 구조 v3 무손실 압축은 선택형 프롬프트 캐싱과 별개이며 계속 활성 상태를 유지한다.

## 성능 검증

- 시작 완료 전 호스트 브리지 호출: 1.2.7의 59회에서 1.2.8의 16회로 감소 (`72.88%`).
- 시작 완료 전 플러그인 인자 호출: 40회에서 0회로 감소. 40개 인자는 시작 완료 후 병렬로 조회한다.
- 동일한 인공 호스트 지연 시험 중앙값: 909.1ms에서 245.1ms로 감소 (`73.04%`). 절대 시간은 시험용 지연값이며 실제 환경에서는 WebView와 호스트 브리지 성능에 따라 달라진다.
- 인자당 10ms 지연 시험에서 플러그인 시작 완료 시 인자 호출은 0회였고, 지연 동기화에서 40개가 함께 처리됐다.
- 신규 기본 OFF 상태에서 `getRootDocument` 호출은 0회였다.

## 남은 위험

- 플러그인 파일 파싱 비용은 1.2.7과 비슷하게 남는다. 이번 핫픽스는 호스트 브리지 대기와 런처 초기화가 플러그인 활성 완료를 막지 않도록 수정한다.
- 기존 설정에서 으아악씨나 모시모씨가 ON이면 시작 완료 후 해당 런처를 정상적으로 생성하므로 OFF 사용자보다 추가 DOM 작업이 발생한다.
